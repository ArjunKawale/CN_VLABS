DNS Resolution Simulator

Interactive simulator for understanding how DNS resolves domain names to IP addresses, including recursive queries, iterative lookups, caching, and the full resolver chain.

Project Overview

Domain Name System (DNS) is the internet's phonebook — it translates human-readable domain names like www.example.com into machine-readable IP addresses like 93.184.216.34. This simulator demonstrates the full DNS resolution process, stepping through each server in the lookup chain: the Recursive Resolver, Root Server, TLD Server, and Authoritative Server.

DNS Theory Summary

  • DNS maps domain names to IP addresses using a hierarchical distributed database.
  • Resolution involves up to four server types: Recursive Resolver, Root, TLD, and Authoritative.
  • DNS uses UDP port 53 for queries (TCP for large responses).
  • Responses are cached with a TTL (Time To Live) value.
  • Root servers know the TLD servers; TLD servers know the Authoritative servers.
  • There are 13 root server clusters managed by IANA globally.

DNS Theory

1. Introduction to DNS

The Domain Name System (DNS) is a hierarchical, distributed naming system for computers and services connected to the Internet or private networks. It translates domain names into IP addresses needed for locating and identifying devices and services using underlying network protocols.

2. DNS Hierarchy

3. DNS Query Types

  • Recursive Query: Client asks resolver to do all the work and return a final answer.
  • Iterative Query: Resolver asks each server in turn; each server refers to the next.
  • Non-recursive Query: Resolver already has the answer cached.

4. DNS Resolution Steps for www.example.com

  1. Client (browser) sends a query to the Recursive Resolver.
  2. Resolver queries a Root DNS Server.
  3. Root Server responds with the address of the .com TLD server.
  4. Resolver queries the .com TLD Server.
  5. TLD Server responds with the Authoritative Name Server for example.com.
  6. Resolver queries the Authoritative Server.
  7. Authoritative Server returns the IP address (e.g., 93.184.216.34).
  8. Resolver caches and returns the IP to the Client.

5. DNS Record Types

  • A Record: Maps domain to IPv4 address.
  • AAAA Record: Maps domain to IPv6 address.
  • CNAME: Canonical Name — alias of another domain.
  • MX Record: Mail exchange server for the domain.
  • NS Record: Name server for the domain.
  • TTL: Time To Live — how long a record is cached.

6. DNS Caching

DNS responses are cached by the resolver and browser. The TTL value (in seconds) determines how long a record is stored before re-querying. Caching reduces latency and load on DNS servers.

7. DNS Security (DNSSEC)

DNSSEC (DNS Security Extensions) adds cryptographic signatures to DNS records to protect against cache poisoning and man-in-the-middle attacks. It verifies that DNS responses are authentic.

Resolution Flow Description

  1. IDLE: No query in progress. Browser has not made a DNS request.
  2. CLIENT QUERY: Browser requests IP for a domain from the Recursive Resolver.
  3. CACHE CHECK: Resolver checks its local cache. If found, skips to DELIVERED.
  4. ROOT QUERY: Resolver asks a Root Server which TLD server to contact.
  5. TLD QUERY: Resolver asks the .com TLD Server for the Authoritative server.
  6. AUTH QUERY: Resolver asks the Authoritative Server for the final IP address.
  7. IP RETURNED: Authoritative Server returns the A record (IP address).
  8. CACHED: Resolver caches the response for future queries (TTL applied).
  9. DELIVERED: Resolver sends the IP address back to the Client browser.
  10. NXDOMAIN: Domain does not exist — error returned to client.

DNS Use Cases

  • Translating domain to IP
  • Mail server lookup (MX)
  • Load balancing via multiple A records
  • CDN routing
  • Service discovery in microservices

HTTP Use Cases

  • Loading web pages
  • REST API communication
  • File downloads
  • Streaming media
  • Web application data exchange
